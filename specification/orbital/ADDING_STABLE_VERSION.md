## 2. Key Files Overview

### `main.tsp` — Service Definition

```typespec
@useAuth(OAuth2Auth<[...]>)
@service(#{ title: "Microsoft Planetary Computer Pro" })
@versioned(Versions)
namespace Microsoft.PlanetaryComputer;
```

- Imports all route files (STAC, Tiler, SAS, InMa)
- Defines auth (OAuth2 implicit flow), service title, and versioning

### `models.common.versions.tsp` — Version Enum

```typespec
enum Versions {
  v2025_04_30_Preview: "2025-04-30-preview",
}
```

**To add a stable version**, add a new entry here (e.g., `v2026_04_15: "2026-04-15"`).

### `client.tsp` — SDK Client Customizations

- Groups operations into four client interfaces: `Ingestion`, `Stac`, `Data`, `SharedAccessSignature`
- Provides `@@clientName` overrides for C#, Python, JavaScript
- Configures `@@convenientAPI`, `@@alternateType`, `@@operationGroup`
- Uses `@useDependency(Versions.v2025_04_30_Preview)` — **must be updated for new version**

### `tspconfig.yaml` — Emitter Configuration

Configures output for: AutoRest (OpenAPI), Python, C#, TypeScript, Java SDKs.

### `data-plane/readme.md` — AutoRest Tags

Each version requires a tag block:
```yaml
tag: package-2025-04-30-preview
```
**Must add a new tag** for stable version.

## Phase 1 — Create Stable Version with Existing Endpoints

### Step 1: Update the Version Enum

In `models.common.versions.tsp`, add the new stable version:

```typespec
enum Versions {
  v2025_04_30_Preview: "2025-04-30-preview",
  v2026_04_15: "2026-04-15",
}
```

### Step 2: Create Example Files for Stable Version

1. Copy the example folder:  
   `cp -r examples/2025-04-30-preview/ examples/2026-04-15/`
2. Update `api-version` values in all example JSON files from `"2025-04-30-preview"` to `"2026-04-15"`.

### Step 3: Update `client.tsp`

Update the `@useDependency` decorator to reference the new stable version:

```typespec
@useDependency(Versions.v2026_04_15)
```

### Step 4: Update `data-plane/readme.md`

Add a new tag block and set it as the default:

```yaml
tag: package-2026-04-15

### Tag: package-2026-04-15

These settings apply only when `--tag=package-2026-04-15` is specified on the command line.

```yaml $(tag) == 'package-2026-04-15'
input-file:
  - Microsoft.PlanetaryComputer/stable/2026-04-15/openapi.json
```

### Step 5: Compile and Validate

```bash
cd specification/orbital/Microsoft.PlanetaryComputer
tsp compile .
```

This generates `data-plane/Microsoft.PlanetaryComputer/stable/2026-04-15/openapi.json` and copies examples.

### Step 6: Verify Output

- Confirm `stable/2026-04-15/openapi.json` exists and contains the correct version string
- Confirm `stable/2026-04-15/examples/` contains all example files
- Verify no regressions in `preview/2025-04-30-preview/` output

## Phase 2 — Add Missing API Endpoints

### Adding an Endpoint

1. **Define models** in the appropriate `models.*.tsp` file
2. **Define routes** in the appropriate `routes.*.tsp` file
3. **Use `@added(Versions.v2026_04_15)`** decorator to mark new endpoints as stable-only
4. **Add the operation** to the appropriate client interface in `client.tsp`
5. **Create example JSON** in `examples/2026-04-15/` (NOT in the preview folder)
6. **Compile and validate**

### Versioning Decorator Pattern

For endpoints that exist only in the stable version:

```typespec
import "@typespec/versioning";
using TypeSpec.Versioning;

@added(Versions.v2026_04_15)
op newOperation(...): Response;
```

For models that exist only in the stable version:

```typespec
@added(Versions.v2026_04_15)
model NewModel {
  prop1: string;
}
```

## Phase 3 — Add New API Endpoints

Same process as Phase 2. New endpoints should:
- Follow existing naming conventions (e.g., `Interface.operationName`)
- Include complete example JSON files
- Be added to the client interface groupings in `client.tsp`
- Have SDK overrides in `overrides.tsp` if they have complex query parameters

## Phase 4 — Push Changes & Create PR

1. Run `tsp compile .` one final time
2. Verify all generated output is correct
3. Create a feature branch: `git checkout -b feature/stable-2026-04-15`
4. Commit all changes (TypeSpec source + compiled output)
5. Push and create PR against main
6. PR will trigger CI validation (OpenAPI linting, breaking change detection, etc.)

## Common Pitfalls

| Pitfall | Solution |
|---------|----------|
| Forgetting to update `api-version` in example files | Use `sed` or script to batch-update |
| Not updating `client.tsp` `@useDependency` | Add new version reference |
| Not updating `readme.md` tags | Add new tag block for stable version |
| Modifying preview version files | Never modify existing preview — only add stable |
| Missing `@added` decorator on new-only endpoints | Use `@added(Versions.v2026_04_15)` |
| Breaking changes in stable vs preview | Stable should be a superset of preview |
| Not running `tsp compile` before committing | Always compile to regenerate OpenAPI output |

## Example JSON File Format

Each example file follows this structure:

```json
{
  "title": "OperationName_Action",
  "operationId": "InterfaceName_OperationName",
  "parameters": {
    "api-version": "2026-04-15",
    "param1": "value1"
  },
  "responses": {
    "200": {
      "body": { ... }
    }
  }
}
```

- `title` and `operationId` must match the TypeSpec operation
- `api-version` must match the target version
- Response body should contain realistic sample data
